Heuristic Evaluation

To clear the evaluation bias, each group member has conducted the heuristic evaluation separately. We have then combined the results.

Visibility of system status

Heuristic Description: The interface should be providing clear feedback for the user when needed. The system should include visual and auditory feedback that clarifies for the user the state of the game. For example, the SLP should know if the exercise was saved or not and the child should know whether he answered correctly or not.

Our Evaluation:

· Drag & drop should show the object being dragged real time. Whenever the uer drags the object, the object does not follow user’s touch on the screen, so it does not give the feeling of being dragged. This is an important visual feedback to have when you have drag & drop functionality.

· If the user spends more time trying to understand the interface, play the question audio again. If the child is confused and is trying to find the answer, there is no audio feedback to ask the question again. Since the child who uses this system might not know how to read and write, this audio feedback is important to have. The question should be reminded to the user, not just visually but also in audio.

Match between system and the real world

Heuristic Description: The system should communicate using an appropriate language for the user. It should follow real world conventions.

Our Evaluation:

The game is intended for both French and English user. For English interface, both the gameplay words and the UI are in English. Therefore, there wasn’t any resultant confusion. However, when settings are changed into French, only the gameplay words are translated into French. The game UI (Menu, Textview and lists) remained in English. For dominate francophones, this may lead to confusions in navigation through the interfaces

Aesthetic and minimalist design

Heuristic Description: Windows contain information that is relevant and needed. There is no information exposed that diminishes the visibility of the important info

Our Evaluation:

· User interface is very simple. Navigation to a different page can be easily completed. However, in the page “prepare exercise” the text and action buttons may be a little too small.

· In general, the UI elements should be bigger. The user is having trouble pressing on small items. Considering Fitts’ Law, this is an important factor in usability.

Simplicity of the design

Heuristic Description: The design of the interface is found to be clear and simple to use. There is no ambiguity in the meaning of the buttons or windows. Also, the positions in the interface make it easy for the user.

Our Evaluation:

· There is an easy flow in the design of buttons. The gameplay itself is very intuitive.

· The autocomplete box is irrelevant and never used. We have observed that no one ever attempted to use the autocomplete box to find words. This complicates the design and should be eliminated or should be implemented in the word list box in a seamless way.

· The interface to prepare an exercise is a bit complicated. The number of UI elements on the screen can be decreased by following a step-by-step approach instead of showing everything at once.


Encouraging the User to Learn

Heuristic Description: The user is motivated by the game and is able to clearly understand the errors that he has made and through the visual or audio animation of the game the user is excited to continue playing.

Our Evaluation:

· The game should be continuous not per exercise based. Currently, after an exercise is over, the game ends there and goes back to start screen. These are very simple exercises and takes very little time to complete. Preparing a new exercise on the other hand takes a long time. Exercises should keep coming to keep the user engaged. So, we believe that there should be a way to bundle the exercises to keep the excitement of the user high.

· The gameplay is speech directed. Any mistakes are caught immediately. Therefore, the user will stay on one level if he/she doesn’t get the right answer. The audio feedback in the gameplay is good.

· To encourage the children to continue, there can be exciting audio supported visuals. Like clapping sound with a cartoon character clapping the child playing the game.

Test Plan Critique

There were 3 main challenges regarding this testing session. Addressing usability goals, improvements made to the prototype and main flaws will be explained in each section.

1. Invalid Tests

There were some tests that were invalid because the prototype was not able to process them (see the second section below) and some that didn’t address the usability goals.

For example, having a statement like “start the application” does not make a valid test because we are only able to test the design with the boundaries of our own creation. Starting the application has nothing to do with the application itself and it has more to do with the operating system being used by the test device.

Tests deemed invalid were not executed.

2. Incomplete Prototype

There were certain aspects of prototypes that wasn’t functioning. Although some of them could be overcame by using Wizard of Oz Techniques, some couldn’t be overcome and effected the testing session.

The testers (parent and child) were francophones; however, the French version of the game was not functioning. To amend for the technical shortcomings, the demonstrator used the wizard of Oz technique to carry out testing session. So, the sound of the application was turned off and the evaluator used her own voice every time the user interacted with the prototype.

Another part that was confusing was in the gameplay screen. Due to some bug, whenever the child pressed on a picture, it was being dragged to a different location, overlapping other pictures on the screen and making it hard to play the game.

Also, “Yes/No” and “Does it rhyme” types of exercise were not ready in the prototype and could not be tested.

3. The Language Barrier

Since the child and the parent were francophones, it effected the testing a lot in many different ways. Our friend from Group 3 was there with us, and although she was intended to be a translator only, she ended up having the most of the dialogue with the children. This was natural, and not accounted for, because the therapist naturally wanted to introduce the French speaker among us to the children. It was hard to get the child to talk with 3 different people 2 of which didn’t speak his only language. Same goes with the parent, she had very little or no English. We asked the French speaker evaluator to translate the dialogue as often as possible, but as the parent seemed in a hurry towards the end, we couldn’t get as much translations. One of our group members who was not with us during the session spoke French and she was able to transcribe some of those dialogues in the recorded video.

Another way this effected testing was that the prototype was not ready yet for French and some features could not be tested or tested with extra assistance from the evaluator.

Design Critique

Based on our heuristic evaluation and observations during the usability testing, we have come up with a formative feedback for Group 5 in 3 main areas of improvement for the game.

1. Step-by-step Approach

We believe that the certain interactions with the game leave too much to look at the screen and analyze what is going on. A good example is the “Prepare an Exercise” screen. There are so many UI elements on this screen and the users can easily be confused as proven by the usability testing.

To make this screen more interesting and easy to interact with for the user, a step-by-step approach should be followed. Such approach can eliminate the need to lay out all the UI elements on one screen.

Another area this approach can be implemented is within the gameplay. As explained in the heuristic evaluation, exercises can be bundled together, i.e. one exercise after another, step by step, to increase the excitement in the gameplay for the user.

2. Improvement of UI elements

In general, the UI elements were easy to interact with and visible enough.

However, some UI elements were too small, and the user failed to be able to click on them. Specifically in the “Prepare an Exercise” screen, this was an issue. These elements should be made larger.

Also, the drag & drop action was not very clear. Whenever the user drags a picture on the screen, the object being dragged should follow the finger, not stay in its initial location until being dropped to the final location. This visual feedback can be important in many ways. This feature should be implemented.

3. Focus on Tablet Gestures vs. Keyboard

We have felt that during some parts of the test, there was a confusion on user’s side to whether use the keyboard or the touch screen. This confusion is also reflected on the design of the UI elements. The prototype was able to handle both double click on an object or a drag action on the same object for the word list when the user wanted to prepare an exercise. When two different actions using two different input methods can have the same results on the screen, this might be considered as a flexibility for the advanced user but it’s most likely a confusion for a novice user. Limiting the ways of interaction and designing the user interface for touch actions, or keystrokes/mouse actions is an important decision that needs to be made. And the final product should be presented to the users in that format. Having a half mobile half desktop system can confuse novice users due to its complicated consequences in user interface design.
