Test Plan & Evaluation

Usability Goals

1. The user interface is intuitive
a) Buttons are clear
b) Drag action can be understood

2. User can play the game without giving too much “How-to” information

3. User wastes less time interacting with the UI and more time playing the game

4. User feels encouraged by the system to improve her short term memory

Benchmark Tasks

1. Set up the game

2. Learn how to play

3. Start the game at level 1

4. Play at least 2 levels

5. Continue to next level

6. Replay same level

7. Restart the game

8. Start the game at a different level than 1

NOTE: One aspect we cannot test with this prototype is with the actual people the evaluator knows. The impact of the 'social' aspect of the game can be only tested once Facebook login is implemented. Thus, this test just includes 3 characters the evaluator does not know and their likes: Emir likes Titanic. Raphael likes photography. Alison likes biking.

Evaluation of Tasks

For each task, if the user asks a question or is frustrated and doesn’t know what to do, that counts as 1 point. Tasks with 0 points are successful tasks. Usability of all other tasks should be revised and improved after the testing.

Individual feedback will be evaluated based on the answers on the debriefing session. Any improvements or suggestions will be considered.

Briefing

1. Welcome

2. Ask the participant sign the consent form, let them know this session will be recorded only to improve the project

3. Start filming the test session

4. Introduction to experimenters

5. Introduction to the project
a) McGill University study
b) Short term memory game
c) Includes people that you know, but the people in the content of the prototype are testers.

6. Explain what they will do
a) “We are not testing you, we are testing the prototype!”
b) Ask if they have used a mobile device before and if they have a Facebook account or know what Facebook is
c) If the user is inexperienced, introduce certain UI elements, like buttons and drag action
d) Otherwise, continue with the session asking them to act as if the prototype is their smartphone screen
e) Let them know they will complete 8 tasks without any help, if the user is inexperienced in terms of mobile device usage, a little help in gestures might be provided as a briefing
f) Tell the user “If you are not sure what to do, do what feels right”
g) “Verbalize your thoughts while going through each task”
h) “You can take breaks whenever you want, no need to rush”

7. Give information for the next task on the list

8. Start the task (If the game is over, note user’s score and let them know)

9. Start writing observations about the evaluator’s interaction with the prototype
a) Record when user hesitates to press on some button or feels frustrated with the interface
b) Record when user asks a question about how to use the prototype
c) Record how long each task takes the user to complete

Debriefing

1. Frustration

a) Did you ever feel frustrated when using the prototype? If so, when?

b) Did you understand where to press? Or what to drag? If not, which task was that button in?

c) Did you feel a particular task was hard to accomplish? Which task was it?

2. Encouragement

a) Did you feel encouraged to improve further in the game or did you just want to complete the level and finish playing?

b) Did you care about your score?

3. Suggestions

a) Did you ever think a certain action during gameplay was unnecessary, what was it?

b) Do you have any further suggestions to help us improve the game?

Learnings

The feedback that we received from our evaluator was mixed. There are some great points. For example, she thought our project is intuitive, simple and fast to pick up. As well she thought that game can get addictive. In addition, at the Jewish Rehabilitation Hospital, they have already begun to use iPad with patients. Therefore, the evaluator thought our project is a good idea; 1. because people already know how to use the technology and 2.they are looking for rehabilitation games. However, there are also some changes that can be made.

First of all, our evaluator thought that our game was currently too simple. To accommodate patients of various levels, we should increase the difficulties in our game. Two the suggestions that was made include: to increase the amount of items that one person can be linked to and to increase the amount people that the user will have to remember. When asked whether or not the game should just start with difficult levels, the evaluator explained that it was good to start at a lower level because this will allow for the accommodation of patients of various levels.

Secondly, our evaluator pointed out that our concepts should be changed. The game should not just end because the patient made a mistake. Our project should lean more towards being a learning thing. The patient should be able to drag everyone to their “appropriate” items. Only when the patient is finished with correlating everyone, should screen pop up to show to patient what their score is and what they have connected wrong. To quote our evaluator “they would […] have the other four [correct], which is ok as opposed to if they got them all wrong. The game should be more of a learning experience.

Thirdly, when asked if a score tracker should be included, our evaluator explained that it should be. In therapies, everything should be kept as objectively as possible. Therefore, it would give patients a sense of goal and also show them if they made an improvement or deterioration.

On the subject of whether or the patient would have a Facebook Account, our evaluator noted that it would be best if we can have two approaches to solve the problem. This is because while we assumed that many people would have a Facebook account, there are many people (especially the elderly) who don’t. Therefore, our evaluator explained it would be better if we can have a Facebook login and a simple login, much alike the popular game Candy Crusher. This would allow patients to correlate people that they know with things and also people that they don’t know. In addition, this method would also increase the difficulties in the levels.

Lastly, on the technical side one of our pages was difficult to interpret. Instead of using cliché symbols, we should use more standard and easier to understand symbols when designing pages. Therefore, one of our pages where we used three circles to indicate page number, our evaluator thought that was not easily understood. Instead we should use arrows to indicate that there more pages to read.

Outcomes

What will we change in our prototype?

1. Let the Player Finish

We have decided to let the player finish until the last matching and then display the game over screen. This way the user will have the chance to answer all of the matchings and see their mistakes later on.

2. Teach

Based on the evaluator's feedback, we've decided to show the user how they have failed, what were the correct matchings after the end of the game. This way we will teach the user and turn the facts about their social contacts into an enjoyable experience by doing so.

3. Score vs. Level

We've decided that scoring should be independent from level, as in the current prototype, the latest level the user has approached before failing is their score. We should let the user continue until they finish all the matchings - even if 1 matching fails - and calculate each correct matching as a +1 point.

4. Gameplay without Facebook

Based on the evaluator feedback, we've decided to allow the users to play the game without Facebook account since some of the old people using the system might not have a Facebook account and it might be a good idea to let people play the game with random characters and their likes as it might be more challenging since they are not related. We will do this by having random datasets of imaginary characters and their likes.
